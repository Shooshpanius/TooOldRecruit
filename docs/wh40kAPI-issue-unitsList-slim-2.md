# Задача для wh40kAPI: оставшиеся проблемы с тяжёлым ответом `/unitsList`

## Статус

**Открыта.** Эта задача создана после того, как предыдущая задача
([`wh40kAPI-issue-unitsList-slim.md`](wh40kAPI-issue-unitsList-slim.md), закрыта в
коммите [`d82a681d`](https://github.com/Shooshpanius/wh40kAPI/commit/d82a681d1051bb49437aa72cda51679e3435d3f1))
устранила часть проблем (`modifierGroups.id/unitId`, `parentId`, пустой `catalogueId`
на дочерних узлах), но ответ `/unitsList` по-прежнему возвращает избыточные данные.

Проверено на фракции Death Guard (`5108-f98-63c2-53cb`):
`GET /api/bsdata/fractions/5108-f98-63c2-53cb/units-list`
Объём ответа: **6.5 МБ** при 5 266 узлах.

---

## Проблемы

### 1. `categories` и `infoLinks` на дочерних узлах (depth ≥ 1)

Клиент TooOldRecruit читает `categories` и `infoLinks` **только у корневых узлов**
(`depth=0`). На дочерних узлах (`depth ≥ 1`) эти массивы никогда не используются при
работе с `/unitsList`.

**Текущее состояние:**
- 309 дочерних узлов содержат непустой массив `categories`
- 461 дочерний узел содержат непустой массив `infoLinks`

**Пример дочернего узла (depth=1), который содержит ненужные категории:**
```json
{
  "id": "13f1-2794-34e7-4bd6",
  "catalogueId": "",
  "name": "Warlord",
  "entryType": "upgrade",
  "categories": [
    { "id": 1166, "unitId": "...", "name": "Warlord", "primary": false }
  ],
  "infoLinks": []
}
```

**Решение:** на всех узлах с `depth ≥ 1` возвращать `"categories": []` и
`"infoLinks": []` (пустые массивы).

---

### 2. Поля `id` и `unitId` в объектах `categories`

Каждый объект в массиве `categories` содержит поля `id` (целочисленный PK базы
данных) и `unitId` (GUID родительского юнита), которые клиент **никогда не читает**.
Клиент использует только `name` и `primary`.

**Текущее состояние:** 1 772 объекта категорий содержат лишние поля `id`/`unitId`.

**Пример:**
```json
{ "id": 1861, "unitId": "5714-846a-d343-d1b4", "name": "Faction: Legiones Daemonicum", "primary": true }
```

**Решение:** в `BsDataUnitNodeLite` использовать slim-проекцию для категорий —
сериализовать только `name` и `primary`:
```json
{ "name": "Faction: Legiones Daemonicum", "primary": true }
```

---

### 3. Поля `id` и `unitId` в объектах `costTiers`

Аналогично: каждый объект в массиве `costTiers` содержит поля `id` и `unitId`,
которые не нужны клиенту. Клиент читает только `minModels`, `maxModels` и `points`.

**Текущее состояние:** 36 объектов `costTiers` содержат лишние поля.

**Пример:**
```json
{ "id": 140, "unitId": "5b4e-dcf1-5c6f-4d35", "minModels": 3, "maxModels": 3, "points": 115.0 }
```

**Решение:** slim-проекция — только `minModels`, `maxModels`, `points`:
```json
{ "minModels": 3, "maxModels": 3, "points": 115 }
```

---

### 4. Поле `points` сериализуется как `decimal` с 28 знаками после запятой

Все значения `points` сериализуются с 28 знаками после запятой:
`0.0000000000000000000000000000` вместо `0`.

**Текущее состояние:** 2 809 узлов имеют подобное значение. Лишний объём: **~74 КБ**.

**Пример:**
```json
"points": 0.0000000000000000000000000000
```

**Решение:** изменить тип поля `Points` в `BsDataUnitNodeLite` с `decimal?` на
`double?` (или `float?`) — это обеспечит компактную сериализацию: `0` вместо
`0.0000000000000000000000000000`.

---

### 5. Поле `id` в объектах внутри JSON-строк `modifiers` и `conditions`

Каждый объект внутри встроенных JSON-строк `modifiers`/`conditions` содержит поле
`"id": null`, которое клиент не использует.

**Текущее состояние:** 6 572 модификатора и 5 985 условий содержат `"id": null`.
Лишний объём: **~135 КБ**.

**Пример:**
```json
"modifiers": "[{\"id\":null,\"field\":\"category\",\"type\":\"set-primary\",\"value\":\"...\"}]"
```

**Решение:** при построении строк `modifiers`/`conditions` не включать поле `id`
(или использовать `[JsonIgnore]`/не проецировать его в DTO).

---

### 6. Pretty-printed JSON (переносы строк и отступы)

Ответ возвращается с полным форматированием (indentation), что добавляет **3.4 МБ**
(52% от общего объёма) только за счёт пробелов и переносов строк.

**Текущее состояние:** объём ответа 6.5 МБ, из которых 3.4 МБ — форматирование.

**Решение (одно из двух):**
- **Вариант A:** Отключить pretty-printing для endpoint `/unitsList` — возвращать
  минифицированный JSON.
- **Вариант B:** Включить HTTP-сжатие (gzip/br) — стандартная middleware для ASP.NET
  Core (`app.UseResponseCompression()`). Сжатие уменьшает repetitive JSON на 80–90%.

---

## Ожидаемый эффект

| Проблема | Примерный объём |
|---|---|
| Pretty-print (пробелы/отступы) | ~3 400 КБ |
| `modifierGroups` (содержимое) | ~1 495 КБ (структурно необходимо) |
| `id`/`null` в modifiers/conditions | ~135 КБ |
| `id`/`unitId` в categories | ~104 КБ |
| `points` с 28 знаками | ~74 КБ |
| `infoLinks` на дочерних узлах | ~18 КБ |
| `categories` на дочерних узлах | ~12 КБ |
| `id`/`unitId` в costTiers | ~2 КБ |

После устранения пунктов 1–5 (без pretty-print) ответ сократится примерно до **1.3 МБ**
(до сжатия), или **~200–350 КБ** при включённом gzip.

---

## Временное решение на стороне TooOldRecruit

До реализации исправлений в wh40kAPI прокси-метод
`BsdataController.GetFractionUnitsList` выполняет **серверную зачистку** ответа:
- убирает `categories` и `infoLinks` из узлов `depth ≥ 1`;
- убирает `id`/`unitId` из объектов `categories` и `costTiers`;
- возвращает минифицированный JSON (без отступов).

Это позволяет уменьшить объём передаваемых данных клиенту до решения задач в wh40kAPI.
После реализации исправлений в wh40kAPI серверная зачистка должна быть удалена —
`BsdataController` вернётся к роли простого прокси.

## Приоритет

Высокий. Текущий объём ответа **6.5 МБ** существенно замедляет загрузку каталога
юнитов для пользователей.
